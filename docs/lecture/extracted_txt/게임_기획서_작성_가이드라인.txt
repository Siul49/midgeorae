01
문제 정의 /  Problem Definition
이 게임이 해결하려는 문제를 명확하게 특정하는 단계
SECTION 01
왜 가장 중요한가?
문제 정의가 흐릿하면 메카닉도, 경험도 방향을 잃습니다. 기획서의 모든 섹션은 이 문제 정의에서 출발합니다.
"이 게임이 왜 존재해야 하는가?"에 대한 대답이 바로 문제 정의입니다.
일상 불편함
매일 겪는 작은 불편, 반복되는 습관
문제
예) 스마트폰 과의존, 수면 부족, 미루
는 습관
사회적 이슈
구조적·사회적 문제를 게임으로 체험
예) 불평등, 기후변화, 전쟁과 민간인, 
노동 문제
감정적 문제
관계, 상실, 소통의 어려움 등 내면의
문제
예) 이별, 부모 자식 관계, 사회적 고립
문제 정의 시 핵심 구분
소재 (Symptom) 스마트폰을 너무 많이 본다
문제 (Problem) 주의 집중 능력의 약화와 자기 통제력 상실
게임 접근 (Game Angle) 매 라운드 선택의 순간, 집중 vs 방해 사이의 딜레마를 체험

01 작성법
문제 정의 — 항목별 작성 가이드
각 칸에 무엇을 어떻게 써야 하는가
01 작성법
① 핵심 문제 (1문장)
작성 포인트
주어(누가) + 상황(어떤 환경에서) + 문제(무슨 문제) 구조로 1문
장 작성.
막연한 주제(스마트폰)가 아닌, 구체적 상황 속 문제를 기술.
✓ 좋은 예시
직장인이 집중해야 할 업무 중 스마트폰 알림에 반복적
으로 반응해 집중력을 잃어간다
✗ 나쁜 예시
스마트폰 중독이 심각하다.
② 문제 발생 맥락
작성 포인트
언제, 어디서, 어떤 상황에서 이 문제가 발생하는지 '장면'을 서
술.
독자가 머릿속에서 그 장면을 그릴 수 있어야 함.
✓ 좋은 예시
회의 중 진동이 울릴 때, 시험 전날 밤, 수업 시간 등 집중
이 필요한 순간
✗ 나쁜 예시
언제 어디서나 발생한다.
③ 플레이어 대상
작성 포인트
"사람들"처럼 모호한 대상이 아닌, 이 문제를 실제로 겪는 특정
집단.
연령, 상황, 성향 등 1~2가지 특성으로 한정.
✓ 좋은 예시
집중 업무가 많은 20~30대 직장인 / 시험 준비 중인 대학생
✗ 나쁜 예시
스마트폰을 사용하는 모든 사람

02
플레이어 행동 설계 /  Player Action Design
플레이어가 게임 안에서 실제로 '무엇을' 하는지를 설계
SECTION 02
Gameplay = 플레이어에게 주어진 선택의 결과 — Sid Meier: "A game is a series of interesting choices"
① 핵심 행동 동사
플레이어가 반복적으로 수행하는 행동
을 명사가 아닌 동사로 서술합니다.
게임 디자이너는 메카닉으로 '문장(플레
이)'을 만들며, 동사가 그 핵심입니다.
예시
"카드를 낸다"가 아닌 → "상대방의 패
를 예측하고 전략적으로 카드를 선택
한다"
② 선택과 딜레마
흥미로운 선택의 3조건: 결과가 불확실
함 / 결과에 의미 있는 차이가 있음 / 전
략적 고민이 필요함
한 선택이 다른 것을 포기하게 만드는
트레이드오프가 있어야 합니다.
예시
"효율을 선택하면 도덕이 희생된다" 
(Train) / "생존을 위해 약탈할지, 굶을
지" (This War of Mine)
③ 성공 / 실패 조건
게임의 목표(Goal)와 종료 조건
(Termination)을 명확히 구분합니다.
규칙은 명확하고 이해하기 쉬워야 하
며, 도전을 통해 즐거움을 획득하게 됩
니다.
예시
"집중 게이지가 0이 되면 게임 오버" / 
"알림 10개를 전부 무시하면 클리어"
✗ 흔한 실수 |  "마우스를 클릭한다", "버튼을 누른다" 같은 입력 동작이 아니라, 게임 세계 안에서의 의미 있는 행동을 서술해야 합니다.

SECTION 
03
핵심 메카닉 설계 /  Core Mechanic Design
행동을 규칙으로 변환하고 반복 가능한 구조를 설계
SECTION 03
메카닉 = 플레이어와 게임의 모든 요소를 연결하는 '행동 인터페이스’  
|  아이디어(명사·형용사·동사) → 규칙 → 도전 → 메카닉
핵심 메카닉 1문장
게임에서 가장 중요한 규칙 1개를 '플레이어는 [행동]을 통해 [목
표]를 달성한다' 형식으로 서술
이 문장이 문제 해결과 직접 연결되어야 합니다.
1개 규칙에만 집중 (2개 이상 서술 금지)
문제 해결 방식이 메카닉 안에 내포됨
플레이어 행동과 목표가 명확히 서술됨
행동 → 규칙 변환
플레이어의 자연스러운 행동을 게임의 공식 규칙으로 바꾸는 과
정
입력 → 시스템 처리 → 출력의 흐름으로 서술.
'만약 ~하면 ~이 된다' 인과 구조
수치화 가능 여부 확인 (HP, 게이지 등)
규칙이 플레이어에게 전달 가능한가
반복 루프 구조
한 사이클(Loop)의 흐름을 단계별로 서술
Micro Loop(매 순간)과 Macro Loop(세션 전체)를 구분해 생각
자극 → 선택 → 결과 → 다음 자극
루프가 무한 반복 가능한지 검증
루프가 플레이어를 '계속 하게' 만드는가
문제 해결 연결
이 메카닉을 플레이하면 어떻게 문제가 해결(또는 체험)되는가?
실제 효과가 없더라도 '경험으로서' 문제를 다루어야 함.
메카닉과 문제 정의가 논리적으로 연결
플레이어가 문제를 '직접 체험'하는가
보조 메카닉이 핵심을 지원하는가

03 심화
메카닉 — 루프 구조 설계법
반복 가능한 게임 루프를 어떻게 설계하는가
03 심화
핵심 게임 루프 (Core Loop) 구조
STEP 1
자극 / 입력
플레이어가 마주하는 상황·사건
→
STEP 2
선택 / 행동
플레이어의 반응과 의사결정
→
STEP 3
결과 / 피드백
시스템의 즉각적인 반응
→
STEP 4
보상 / 연결
다음 루프로 이어지는 동기
↺ 반복
레퍼런스 예시 — Papers, Please
자극
입국자가 서류를 제출 →
선택
정보 대조 → 승인 or 거부 →
결과
월급 ±, 처벌 위험, 감정
반응
→
연결
다음 입국자 → 루프 반복
자주 묻는 질문
Q  핵심 메카닉이 2개인데 어떻게 하나요?  |  A  하나를 선택하세요. 이 과제의 핵심은 '핵심 시스템 1개에 집중'입니다.
Q  메카닉이 너무 단순한 것 같아요.  |  A  단순함이 나쁜 것이 아닙니다. 중요한 것은 메카닉이 문제 해결과 직결되는가입니다.

04
목표 경험 정의 /  Target Experience
플레이어가 이 게임을 통해 무엇을 느끼고 깨달아야 하는가
SECTION 04
목표 경험의 3축
감정적 경험 Emotional
긴장, 성취, 몰입, 죄책감, 공감 등
'플레이 중 어떤 감정을 느끼는가?'
작성 포인트
단순히 '재미있다'가 아닌, 구체적인
감정 단어 사용.
예) '선택 직전의 긴장감'이나 '결과를
봤을 때의 죄책감'
인지적 경험 Cognitive
학습, 이해, 관점 전환, 인식 변화
'플레이 후 무엇을 알게 되는가?'
작성 포인트
게임이 가르치려는 통찰을 명확히.
예) '집중이 얼마나 쉽게 깨지는지를
직접 느낀다'
행동적 변화 Behavioral
습관 개선, 태도 변화, 사회적 공감
'게임 이후 무엇이 달라지는가?'
작성 포인트
게임 밖에서의 변화를 염두에 둠.
예) '게임 후 실제로 폰을 잠그는 행동
증가'
경험의 흐름 /  Emotional Arc
호기심 → 집중 → 압박↑ → 딜레마 → 결말 충격
핵심 기준 자가 체크
|  '이 게임은 어떤 문제를 어떻게 해결하는가?' + '어떤 딜레마를 경험하는가?' — 반드시 한 문장씩 완성할 것

자주 하는 실수 — 감점 요인 총정리
Common Mistakes  |  기획서 제출 전 이 슬라이드로 자가 점검하세요
01 문제 정의
✗ 문제가 너무 광범위하거나 막연함
→  '스마트폰 과의존' → '업무 중 집중력 손실'처럼 상황을 구체화
01 문제 정의
✗ 소재(증상)를 문제로 오인
→  증상이 아닌 그 이면의 본질적 문제(원인·구조)를 서술
02 행동 설계
✗ 행동을 명사로 서술 (예: '전투')
→  동사 중심으로 재서술 (예: '적의 패턴을 파악하고 공격 타이밍
을 결정한다')
02 행동 설계
✗ 선택지가 하나뿐 (딜레마 없음)
→  A와 B 중 하나를 선택할 때 각각 잃는 것이 무엇인지 명시
03 메카닉
✗ 메카닉을 2개 이상 서술
→  반드시 핵심 1개만 선택, 나머지는 보조 메카닉으로 분류
03 메카닉
✗ 메카닉이 문제 해결과 무관
→  '이 메카닉을 플레이하면 문제가 어떻게 체험되는가?'를 한
문장으로 서술
04 경험
✗ '재미있다'만 서술 (감정 미구체화)
→  '결말에서 충격을 받는다', '선택 직전 긴장감'처럼 구체적 감
정 서술
05 레퍼런스
✗ 레퍼런스 게임만 설명하고 연결 없음
→  반드시 '우리 게임에서 어떻게 활용하는가'를 서술해야 의미
있음
