# GAME-6 물품 데이터 확장 설계

## 배경

`GAME-6`은 거래 공개, 수집가, 리셀러, 숨은 위험 정보 시스템이 작동할 수 있도록 물품 데이터를 먼저 정리하는 기반 작업이다. 이번 작업은 거래 플로우나 공개 규칙을 바꾸지 않고, 이후 이슈가 사용할 수 있는 데이터와 서버 스냅샷 기반을 만든다.

## 확정 범위

- 일반 물품은 24개로 확장한다.
- 카테고리는 `electronics`, `fashion`, `hobby`, `living` 4개를 유지하고 각 6개씩 배치한다.
- 상태는 4단계로 고정한다.
  - `mint`: 민트급
  - `used`: 사용감 있음
  - `defective`: 하자 있음
  - `broken`: 파손
- 위험 물품은 24개 중 6개로 둔다.
  - `defective` 4개
  - `broken` 2개
- `mint`와 `used`는 정상 거래 물품으로 본다.
- `defective`와 `broken`은 추리와 검수 대상이 되는 위험 물품으로 본다.
- 기존 벽돌 카드는 유지한다. `broken`은 실물은 있지만 가치가 크게 떨어지는 물품이고, 벽돌은 사기성 0원 카드로 구분한다.

## 데이터 모델

기존 `Item` 데이터는 최소한 다음 정보를 표현할 수 있어야 한다.

- `id`: 안정적인 영문 식별자
- `name`: 화면에 표시할 한국어 물품명
- `emoji`: 기존 보드게임 UI용 간단 표시
- `category`: `electronics | fashion | hobby | living`
- `marketPrice`: 거래 기준가
- `condition`: `mint | used | defective | broken`

기존 코드의 `basePrice` 사용처가 아직 남아 있으므로, 구현 단계에서는 호환성을 확인해 `marketPrice` 중심으로 바꾸되 필요한 경우 좁은 범위의 호환 처리를 둔다.

`acquiredPrice`는 정적 물품 정보가 아니라 카드 인스턴스가 획득된 가격이다. 시작 손패는 값이 없을 수 있고, 거래 성사 후 구매자 카드에 거래가를 기록할 수 있게 준비한다.

## 서버 스냅샷

서버 카드 스냅샷은 공개 규칙을 바꾸지 않은 채 새 데이터를 운반할 수 있어야 한다.

- 소유자에게 보이는 내 손패에는 물품명, 카테고리, 기준가, 상태가 포함된다.
- 일반 거래 중 뒤집힌 물건은 기존처럼 숨김 상태를 유지한다.
- `directTrade`처럼 공개 거래가 필요한 흐름은 기존 공개 여부 판단을 따른다.
- 위험 여부는 이번 이슈에서 새로 공개하지 않는다. 실제 공개 정책은 `GAME-1`에서 다룬다.

## 카드 자산

현재 17번 이후 새 물품 SVG가 없을 수 있다. 이번 이슈에서는 UI가 깨지지 않도록 다음 중 하나를 사용한다.

- 기존 생성 스크립트로 새 물품 SVG를 재생성한다.
- 이미지가 없는 경우 기존 카드 뒷면 또는 기본 물품 아이콘 fallback을 유지한다.

고품질 카드 일러스트 재작업은 이번 범위에 포함하지 않는다.

## 테스트와 완료 기준

완료 기준은 다음과 같다.

- `ALL_ITEMS`가 24개이고, 4개 카테고리가 각각 6개씩 존재한다.
- 상태 분포가 `mint` 6개, `used` 12개, `defective` 4개, `broken` 2개다.
- 5인 게임 시작 시 각 플레이어가 5장의 물품 카드를 받아도 일반 물품 풀이 즉시 반복되지 않는다.
- 서버 스냅샷이 새 필드를 타입 안정적으로 포함한다.
- 기존 거래 공개 규칙은 바뀌지 않는다.
- 검증 기본값은 `pnpm test`, `pnpm lint`, `pnpm build`다.
