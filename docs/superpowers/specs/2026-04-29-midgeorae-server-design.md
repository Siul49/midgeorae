# 믿거래 서버형 멀티플레이 설계

## 배경

`믿거래`는 중고거래의 정보 비대칭, 비매너 거래, 평판 불신을 사회추론과 거래 협상으로 체험하게 하는 4인 웹 보드게임이다. 현재 프로젝트에는 Next.js 기반의 단일 기기용 게임 초안이 있으며, 목표는 한 기기 제약을 없애고 각 플레이어가 자기 기기로 접속해 바로 플레이할 수 있게 만드는 것이다.

## 승인된 방향

Cloudflare Tunnel을 통해 호스트의 노트북에서 실행 중인 Next.js 서버를 인터넷에 공개한다. 플레이어는 공개 URL과 방 코드로 접속한다. 게임 상태는 우선 노트북 서버 메모리에 저장하되, 상태 전이와 이벤트 처리를 독립 모듈로 분리해 이후 Supabase 저장소로 옮기기 쉽게 만든다.

이 방식은 빠른 플레이 가능성과 향후 확장성을 함께 잡는다. 서버를 끄면 진행 중인 방은 사라지지만, 수업 발표나 팀 테스트처럼 짧은 세션에는 충분하다.

## 핵심 플레이 경험

호스트는 `/game`에서 새 방을 만들고 방 코드와 접속 URL을 공유한다. 네 명의 플레이어가 각자 브라우저에서 이름을 입력하고 입장한다. 네 명이 모이면 호스트가 게임을 시작하고, 서버가 시민 3명과 빌런 1명을 무작위로 배정한다.

각 플레이어 화면에는 본인 역할, 본인 미션, 본인 선택지가 표시된다. 공용 정보 영역에는 방 상태, 현재 턴, 공개 보드, 공개 로그, 플레이어별 공개 자산과 매너 온도가 표시된다. 빌런의 정체와 비밀 미션은 해당 플레이어에게만 보인다.

## 게임 흐름

1. 호스트가 방을 생성한다.
2. 플레이어 4명이 방 코드로 입장한다.
3. 호스트가 게임을 시작한다.
4. 서버가 역할과 빌런 미션을 배정한다.
5. 라운드와 턴이 진행된다.
6. 현재 턴 플레이어가 주사위, 구매, 판매, 거래 제안, 이벤트 처리를 수행한다.
7. 거래 대상 플레이어는 자기 화면에서 거래 참여, 가격 제안, 수락, 거절을 선택한다.
8. 거래 후 관련 플레이어가 평가를 남긴다.
9. 최대 라운드 종료 또는 즉시 종료 조건 발생 시 빌런 투표를 진행한다.
10. 서버가 빌런 색출 여부, 미션 성공 여부, 자산 점수를 계산해 결과를 공개한다.

## 서버 구조

초기 구현은 Next.js Route Handler 기반의 가벼운 인메모리 서버로 구성한다.

- `RoomStore`: 방 목록, 플레이어 세션, 현재 게임 상태를 메모리에 보관한다.
- `GameEngine`: 순수 함수 중심으로 게임 상태 전이를 처리한다.
- `Room API`: 방 생성, 입장, 시작, 액션 제출, 현재 상태 조회를 담당한다.
- `Realtime Channel`: 4인 동기화용 상태 갱신 알림을 제공한다.

실시간 동기화는 첫 버전에서 Server-Sent Events 또는 짧은 폴링으로 구현한다. 플레이어 수가 4명이고 턴 기반 게임이므로, WebSocket 전용 서버 없이도 충분히 안정적으로 동작한다.

## 클라이언트 구조

클라이언트는 크게 세 화면으로 나눈다.

- 로비 화면: 방 생성, 방 코드 입력, 플레이어 이름 입력, 접속 URL 안내.
- 플레이 화면: 공개 보드, 현재 턴, 플레이어 요약, 개인 역할/미션, 현재 액션 패널.
- 결과 화면: 빌런 정체, 투표 결과, 점수표, 승리 조건 설명.

호스트는 방 시작 권한을 가진다. 게임 중 핵심 액션은 현재 턴 플레이어 또는 해당 액션 대상자만 제출할 수 있게 서버에서 검증한다.

## 데이터 모델

주요 모델은 다음 경계를 가진다.

- `Room`: 방 코드, 생성 시각, 상태, 호스트 플레이어 ID, 플레이어 목록, 게임 상태.
- `PlayerSession`: 플레이어 ID, 이름, 호스트 여부, 접속 토큰.
- `GameState`: 라운드, 턴, 보드 위치, 플레이어 공개/비공개 상태, 거래 상태, 로그, 투표 상태.
- `GameAction`: 주사위 굴림, 구매, 판매 시작, 거래 참여, 가격 제안, 수락/거절, 평가, 투표.

서버는 플레이어별 응답을 필터링한다. 예를 들어 시민에게는 빌런 정보가 보이지 않고, 빌런에게만 비밀 미션을 내려준다.

## 오류 처리

방 코드가 없거나 만료된 경우 로비로 돌려보내고 재입장을 안내한다. 이미 4명이 찬 방은 추가 입장을 막는다. 잘못된 턴 액션, 권한 없는 액션, 금액 부족 거래, 종료된 방의 액션은 서버에서 거부한다.

네트워크 연결이 끊긴 플레이어는 같은 브라우저 세션 토큰으로 재접속할 수 있다. 첫 버전에서는 서버 재시작 후 복구는 지원하지 않는다.

## 테스트와 검증

구현 후에는 최소 네 개의 브라우저 세션으로 다음 흐름을 확인한다.

- 방 생성과 4인 입장.
- 호스트 시작 후 역할이 플레이어별로 다르게 보이는지 확인.
- 턴 진행, 주사위, 구매/판매, 거래 제안/수락/거절.
- 거래 후 평가와 매너 온도 변화.
- 최종 빌런 투표와 결과 계산.
- Cloudflare Tunnel URL을 통한 외부 접속.

게임 로직은 가능한 한 순수 함수로 유지해 단위 테스트를 붙인다. 서버 API는 잘못된 권한과 잘못된 순서의 액션을 거부하는지 확인한다.

## 범위 제외

첫 버전에서는 계정 로그인, 영구 저장, 관전 모드, 게임 재개용 DB 저장, 공개 매치메이킹, 채팅, 결제, 관리자 대시보드는 만들지 않는다. 목표는 한 호스트가 노트북으로 서버를 열고 4명이 인터넷 URL로 접속해 한 판을 끝까지 플레이하는 것이다.
