# 믿거래 이슈별 참고 문서

이 문서는 `Siul49/midgeorae`의 열린 이슈를 다른 세션에서도 바로 이어서 작업할 수 있도록 정리한 참고 문서다. 각 이슈는 작은 브랜치에서 독립적으로 끝내고 병합하는 것을 원칙으로 한다.

## 공통 작업 규칙

- 브랜치는 `codex/`를 쓰지 않는다.
- 기능 브랜치: `feat/game-<issue-number>-<topic>`
- 리팩터링 브랜치: `refactor/game-<issue-number>-<topic>`
- 문서 브랜치: `docs/game-<issue-number>-<topic>`
- 정리 브랜치: `chore/game-<issue-number>-<topic>`
- 각 이슈는 가능한 한 하나의 PR로 끝낸다.
- PR 전 검증 기본값은 `pnpm test`, `pnpm lint`, `pnpm build`다.

## 이슈 작업 순서

1. [#1 GAME-0A](https://github.com/Siul49/midgeorae/issues/1): 게임 영역 경계와 폴더 구조 정리
2. [#2 GAME-0B](https://github.com/Siul49/midgeorae/issues/2): 게임 규칙과 서버 액션 처리 책임 분리
3. [#3 GAME-0C](https://github.com/Siul49/midgeorae/issues/3): 생성물, 로그, 문서 위치와 추적 정책 정리
4. [#4 GAME-6](https://github.com/Siul49/midgeorae/issues/4): 물품, 카테고리, 상태 데이터 확장
5. [#5 GAME-9](https://github.com/Siul49/midgeorae/issues/5): 판매자 중심 거래 신청을 구매 주도 행동 카드로 재설계
6. [#6 GAME-1](https://github.com/Siul49/midgeorae/issues/6): 거래 품목 공개와 숨은 위험 정보 보완
7. [#7 GAME-2](https://github.com/Siul49/midgeorae/issues/7): 8턴 고정 규칙을 시장 진행도 기반 종료 조건으로 재설계
8. [#8 GAME-5](https://github.com/Siul49/midgeorae/issues/8): 평판 토큰을 신뢰도 시스템으로 재설계
9. [#9 GAME-3](https://github.com/Siul49/midgeorae/issues/9): 시민 직업 규칙 문서화와 구현
10. [#10 GAME-7](https://github.com/Siul49/midgeorae/issues/10): 빌런 행동 미션과 증거 시스템 추가
11. [#11 GAME-8](https://github.com/Siul49/midgeorae/issues/11): 최종 결과 점수표와 승리 조건 정리
12. [#12 GAME-4](https://github.com/Siul49/midgeorae/issues/12): 다른 와이파이에서도 접속 가능한 배포 경로 확정

## 이슈별 참고

### GAME-0A: 영역 경계와 폴더 구조

목표는 폴더 이동 자체가 아니라 이후 작업 충돌을 줄이는 경계 설계다. 게임은 앞으로 `rules`, `domain`, `server`, `online`, `assets`, `docs/game`으로 나뉜다.

권장 산출물:

- `docs/game/architecture.md`
- 현재 `room-store.ts` 책임 목록
- 이후 이슈들이 어느 폴더를 건드릴지에 대한 표

주의:

- 이 단계에서는 동작 변경을 하지 않는다.
- 테스트 실패가 있으면 고치기보다 현재 상태를 기록한다.

### GAME-0B: 규칙과 서버 액션 책임 분리

목표는 `room-store.ts`를 줄이고, 규칙 계산을 독립 테스트 가능한 함수로 빼는 것이다.

분리 후보:

- 카드/직업/시장 진행도 상수: `src/features/game/rules`
- 거래 완료, 평판 변경, 점수 계산: `src/features/game/domain`
- 방 생성, 액션 라우팅, 스냅샷 필터링: `src/features/game/server`

주의:

- 이슈 0B는 리팩터링이다. 거래 규칙, 평판 규칙, 턴 규칙을 바꾸지 않는다.

### GAME-0C: 생성물과 로그 추적 정책

목표는 기능 PR에 출력물과 로그가 섞이지 않게 하는 것이다.

정리 대상:

- `.codex-*`
- `.playwright-mcp/`
- `outputs/`
- 카드 생성 스크립트와 생성 결과물

판단 기준:

- 사람이 직접 편집하는 원본은 추적한다.
- 명령으로 재생성 가능한 결과물은 원칙적으로 무시하거나 별도 산출물로 둔다.
- 발표/공유용 최종 파일은 보존 근거를 문서화한다.

### GAME-6: 물품/카테고리/상태 데이터 확장

거래 공개, 수집가, 리셀러, 위험 정보 시스템의 기반 이슈다.

권장 데이터:

- `id`
- `name`
- `category`
- `marketPrice`
- `condition`
- `riskTag`
- `acquiredPrice`

카테고리 초안:

- 전자기기
- 패션
- 취미
- 생활

완료 기준:

- 최소 24개 이상 물품
- 5인 게임에서도 카드 분배가 단조롭지 않음
- 같은 카테고리 3개 수집 미션이 억지스럽지 않음

### GAME-9: 구매 주도 거래 신청

현재 흐름은 "내가 팔 물건을 골라 상대에게 제안"한다. 바꿀 방향은 "구매자가 다른 플레이어의 공개 물건에 거래 신청"하는 구조다.

핵심 개념:

- 기존 `sellerId`, `buyerId` 중심 상태를 `requesterId`, `ownerId` 중심으로 재검토한다.
- 액션명은 `sell`보다 `tradeRequest` 또는 `purchaseRequest`가 적합하다.
- 판매자는 수락, 거절, 가격 조정으로 응답한다.

주의:

- 이 이슈는 흥정가 구현 전 기반 작업이다.
- 기존 판매자 중심 테스트는 구매 주도 플로우로 바꿔야 한다.

### GAME-1: 거래 품목 공개와 숨은 위험

목표는 "뭘 거래하는지 모르는 불합리함"을 없애면서도 사기/추리 재미를 유지하는 것이다.

거래 상대에게 공개:

- 물건명
- 카테고리
- 시세
- 제시 가격

기본 비공개:

- 실제 상태
- 하자 여부
- 위장 여부
- 벽돌 여부

검증 규칙:

- 모든 플레이어는 게임 전체 2회 검증 찬스를 가진다.
- 검수자 직업은 이 검증 찬스를 더 강하게 쓰거나 보너스를 받는다.
- 직거래 카드는 숨은 위험 정보를 확인하는 특수 거래 카드로 둔다.

### GAME-2: 시장 진행도와 종료 조건

8턴 고정 대신 시장 진행도로 바꾼다.

초안:

- `marketActionLimit = playerCount * 5`
- `usedActionCount / marketActionLimit`을 시장 진행도로 표시
- 시장 마감 후 최종 투표
- 시장 마감 전 빌런 평판 0이면 시민 조기 승리
- 시장 마감 시 빌런이 미션을 완료하고 정체를 피했으면 빌런 승리
- 빌런이 정체를 피했지만 미션 미완료면 시민 방어 승리

주의:

- 턴을 너무 짧게 잡으면 평판 투표와 미션 수행이 의미 없어지므로 4인 기준 20행동을 1차 기준으로 본다.

### GAME-5: 평판 시스템 개선

현재 "내 평판을 깎아서 남에게 준다"는 구조를 없앤다.

새 규칙:

- 시작 평판 5
- 만족 후기: 대상 평판 +1
- 불만 후기: 대상 평판 -1
- 작성자 평판은 소모하지 않음
- 빌런 평판 0: 시민 조기 승리
- 시민 평판 0: 강제 투표 또는 시장 조기 마감 후보

주의:

- 악성 후기는 GAME-7 증거 시스템과 연결한다.

### GAME-3: 직업 규칙

선행 문서:

- `docs/game/rules/jobs.md`

직업은 시민에게 개인 목표를 주는 장치다. 팀 목표는 여전히 빌런 색출이고, 직업 미션은 보너스와 행동 동기를 만든다.

직업 초안:

- 검수자: 숨은 위험 정보 확인
- 흥정가: 가격 조정 거래 참여
- 수집가: 같은 카테고리 3개 보유
- 신고자: 증거 2개 이상 보유 후 빌런 투표
- 리셀러: 구매가보다 비싸게 2회 재판매

### GAME-7: 빌런 행동과 증거

빌런은 "숨어 있기"만으로 이기지 못하고, 위험 행동을 수행해야 한다.

빌런 행동 후보:

- 위험 물건 거래 성공
- 시세보다 과도하게 비싼 거래
- 반복 거래 취소 유도
- 악성 후기
- 특정 시민 여론몰이

증거 후보:

- 위험 물건 감정 결과
- 시세 괴리 거래 기록
- 반복 취소 기록
- 악성 후기 기록
- 평판 급락 기록

주의:

- 빌런 전용 버튼을 노골적으로 만들면 정체가 바로 드러난다. 일반 행동 안에서 빌런 미션이 달성되게 한다.

### GAME-8: 결과 점수표

팀 승리와 개인 점수를 분리해 보여준다.

결과 화면 구성:

- 시민/빌런 팀 승리
- 빌런 정체
- 빌런 미션 성공 여부
- 개인 자산
- 평판 점수
- 직업 미션 보너스
- 빌런 보너스/패널티
- 최종 개인 순위

주의:

- "시민이 팀으로 이겼지만 개인 1등은 따로 있다"는 구조를 명확히 보여줘야 한다.

### GAME-4: 배포

1차 목표는 다른 와이파이에서도 접속 가능한 데모다.

권장 순서:

1. `pnpm build && pnpm start` 확인
2. Cloudflare Tunnel로 로컬 서버 외부 공개
3. 친구가 다른 와이파이에서 접속 가능한지 확인
4. 장기 배포가 필요하면 Render/Fly + 외부 저장소 검토

주의:

- 현재 방 상태는 서버 메모리에 있다.
- Vercel은 현재 구조와 궁합이 약하다.
- 장기 배포 전에는 Supabase/Redis 같은 외부 저장소 전환이 필요하다.
