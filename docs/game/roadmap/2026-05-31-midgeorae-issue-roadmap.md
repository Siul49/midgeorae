# 믿거래 온라인 게임 이슈 로드맵

작성일: 2026-05-31

## 전제

- 대상 구현은 `/game` 온라인 서버형을 먼저 한다.
- GitHub 이슈 대상 레포는 `Siul49/midgeorae`다. GitHub URL slug는 한글 `믿거래`가 그대로 유지되지 않아 영문 slug를 사용하고, 레포 설명과 문서에서 `믿거래` 이름을 사용한다.
- 현재 로컬 `origin`은 `Siul49/moim`을 가리킨다. 믿거래 레포는 별도 원격 `midgeorae`로 연결한다.
- 브랜치명에는 `codex/`를 쓰지 않는다. 대중적인 브랜치 규칙을 사용한다.
- 권장 브랜치 형식:
  - 기능: `feat/game-<issue-number>-<topic>`
  - 리팩터링: `refactor/game-<issue-number>-<topic>`
  - 문서: `docs/game-<issue-number>-<topic>`
  - 정리/설정: `chore/game-<issue-number>-<topic>`

## 유지보수 구조 제안

현재 `/game`은 게임 규칙, 서버 상태, 카드 데이터, UI, 테스트가 한 기능 폴더 안에 모여 있다. 작은 게임 초안에는 괜찮지만, 앞으로 거래 규칙, 직업, 빌런 행동, 평판, 배포가 각각 커지면 같은 파일을 계속 건드리게 된다. 그래서 Issue 0은 단순 폴더 이동이 아니라 "다른 영역을 수정할 때 충돌을 줄이는 경계 설계"가 목표다.

권장 영역:

- `src/features/game/rules`: 정적 규칙과 밸런스 상수. 직업, 미션, 행동 카드, 시장 진행도, 평판 정책을 둔다.
- `src/features/game/domain`: 순수 판정 로직. 거래 판정, 감정, 직업 미션 진행도, 빌런 미션 진행도, 점수 계산을 둔다.
- `src/features/game/server`: 방 생성, 플레이어 세션, 액션 처리, 스냅샷 필터링, 저장소 어댑터를 둔다.
- `src/features/game/online`: 온라인 화면 UI. 로비, 테이블, 거래 패널, 직업 패널, 투표/결과 패널을 분리한다.
- `src/features/game/assets` 또는 `public/game-cards`: 카드 이미지와 출력물. 생성 스크립트와 결과물을 분리한다.
- `docs/game`: 룰북, 밸런스 메모, 이슈 로드맵, 배포 절차를 둔다.

이 구조의 장점:

- 거래 규칙을 바꿀 때 UI 전체를 열지 않아도 된다.
- 직업/빌런 미션은 `rules`와 `domain` 중심으로 바뀌고, 서버는 결과만 호출한다.
- 배포/저장소 변경은 `server` 저장소 어댑터 중심으로 바뀐다.
- 카드 데이터 추가는 `rules`/`assets` 중심으로 끝난다.
- 다른 세션이 같은 시점에 작업해도 파일 충돌이 줄어든다.

## Issue 0A: 게임 영역 경계와 폴더 구조 정리

제목: `[GAME-0A] 믿거래 온라인 게임 영역 경계와 폴더 구조 정리`

브랜치: `refactor/game-0a-boundaries`

목표:
게임 규칙, 서버 상태, UI, 문서, 생성물을 분리해 이후 이슈들이 서로 같은 파일을 불필요하게 건드리지 않게 한다.

작업:

- `src/features/game/server/room-store.ts`가 담당하는 책임을 목록화한다.
- `rules`, `domain`, `server`, `online`, `docs/game` 기준으로 최종 폴더 구조를 문서화한다.
- 이 단계에서는 대규모 로직 변경 없이 파일 책임과 이동 계획을 명확히 한다.
- 기존 테스트 실행 명령과 실패/성공 상태를 기록한다.

완료 기준:

- `docs/game/architecture.md`에 각 영역의 책임이 설명된다.
- 이후 이슈가 어떤 폴더를 중심으로 작업할지 판단할 수 있다.
- `pnpm test` 기준 현재 상태가 기록된다.

## Issue 0B: 게임 규칙/데이터/서버 액션 책임 분리

제목: `[GAME-0B] 게임 규칙과 서버 액션 처리 책임 분리`

브랜치: `refactor/game-0b-rules-domain`

목표:
거래, 평판, 직업, 빌런 미션 같은 규칙을 서버 방 저장소에서 분리해 테스트 가능한 작은 단위로 만든다.

작업:

- 카드/직업/시장 진행도 상수는 `rules`로 이동한다.
- 거래 완료, 평판 변경, 점수 계산은 `domain` 함수로 분리한다.
- 서버는 액션 유효성 검사, 상태 변경 호출, 스냅샷 생성에 집중한다.
- 기존 동작을 바꾸지 않고 테스트를 유지한다.

완료 기준:

- `room-store.ts`가 방 흐름과 액션 라우팅 중심으로 줄어든다.
- 거래/평판/점수 계산 함수가 독립 테스트 가능해진다.
- 기존 `room-store.test.ts`가 통과한다.

## Issue 1: 거래 품목 공개와 숨은 위험 정보 보완

제목: `[GAME-1] 거래 품목 공개와 숨은 위험 정보 보완`

브랜치: `feat/game-1-item-disclosure`

목표:
거래 품목 자체를 확인할 수 없던 문제를 고친다. 물건명, 카테고리, 시세는 거래 상대에게 공개하되, 상태와 위험 정보는 숨겨 심리전을 유지한다.

규칙:

- 거래 제안 시 공개되는 정보:
  - 물건명
  - 카테고리
  - 시세
  - 제시 가격
- 기본 비공개 정보:
  - 실제 상태
  - 하자 여부
  - 위장 여부
  - 벽돌 여부
- 보유자는 자기 물건의 상태와 위험 정보를 안다.
- 거래 상대는 직거래 카드 또는 검수자 감정 전까지 숨은 위험 정보를 모른다.
- 벽돌은 "벽돌"로 밝혀도 되고, 특정 아이템처럼 위장되는 위험 카드로도 확장할 수 있다.
- UI는 공개 정보와 미확인 위험 정보를 구분해서 보여준다.

작업:

- `ItemCardSnapshot`에 `category`, `condition`, `riskTag`, `publicName` 또는 동등한 필드를 검토한다.
- 일반 거래의 `pendingDealItem`에서 공개 정보는 보이게 한다.
- 숨은 위험 정보는 거래 당사자라도 감정/직거래 전까지 제한한다.
- UI에 "공개 정보"와 "미확인 위험" 섹션을 분리한다.

완료 기준:

- 거래 상대가 물건명/카테고리/시세를 보고 거래 판단을 할 수 있다.
- 상태/위험 정보는 감정 전까지 공개되지 않는다.
- 거래 재미가 단순 정보 비공개가 아니라 "위험 판단"으로 이동한다.

## Issue 2: 시장 진행도와 종료 조건 재설계

제목: `[GAME-2] 8턴 고정 규칙을 시장 진행도 기반 종료 조건으로 재설계`

브랜치: `feat/game-2-market-clock`

목표:
8턴 고정의 임의성을 제거하고, 플레이어 수와 미션 수행 가능성을 고려한 시장 마감 규칙을 만든다.

권장 규칙 초안:

- 시장 진행도는 `사용된 행동 카드 수 / 전체 행동 카드 예산`으로 표시한다.
- 전체 행동 카드 예산은 1차 밸런스값으로 `플레이어 수 x 5`를 사용한다.
- 4인 기준 20번 행동이 가능하므로, 빌런 미션과 시민 직업 미션이 달성될 여지를 준다.
- 시장이 마감되면 최종 투표로 진입한다.
- 시장 마감 시 빌런이 들키지 않았고 빌런 미션도 완료했다면 빌런 승리.
- 시장 마감 시 빌런이 들키지 않았더라도 빌런 미션이 미완료라면 시민 방어 승리로 본다.
- 시장 마감 전 빌런 평판이 0이 되면 시민 조기 승리.

작업:

- `maxRounds` 대신 `marketActionLimit`, `usedActionCount` 개념을 도입한다.
- UI의 `Round` 표시를 `시장 진행도`로 바꾼다.
- 행동 카드 사용이 끝날 때마다 진행도를 증가시킨다.
- 시장 마감 조건과 최종 투표 조건을 서버 테스트로 고정한다.

완료 기준:

- 플레이어 수에 따라 게임 길이가 자동 조정된다.
- 왜 게임이 끝나는지 설명 가능하다.
- 빌런이 숨어만 있어도 이기지 못하고, 미션 수행 압박을 받는다.

## Issue 3: 직업 규칙 문서화와 구현

제목: `[GAME-3] 시민 직업 규칙 문서화와 구현`

브랜치: `feat/game-3-job-rules`

선행 문서:

- `docs/game/rules/jobs.md`

목표:
직업별 컨셉, 능력, 개인 미션, 보상을 먼저 문서화한 뒤 구현한다.

직업 초안:

- 검수자
  - 컨셉: 거래 물건을 꼼꼼히 확인하는 사람.
  - 능력: 게임 전체 2회, 진행 중인 거래 물건의 숨은 위험 정보를 본인만 확인한다.
  - 미션: 감정으로 위험 거래를 1회 이상 피하거나, 감정한 물건으로 정상 거래를 1회 성사한다.
- 흥정가
  - 컨셉: 가격을 흔들어 거래 흐름을 바꾸는 사람.
  - 능력: 가격 조정 제안을 사용할 때 보너스 또는 추가 기회를 갖는다.
  - 미션: 가격 조정이 포함된 거래에 3회 참여한다.
- 수집가
  - 컨셉: 특정 분야 물건을 모으는 사람.
  - 능력: 같은 카테고리 보유 현황을 UI에서 잘 볼 수 있다.
  - 미션: 같은 카테고리 물건 3개를 보유한다.
- 신고자
  - 컨셉: 수상한 행동을 증거로 묶어내는 사람.
  - 능력: 증거 토큰을 더 쉽게 얻거나, 최종 투표에서 증거 보너스를 받는다.
  - 미션: 증거 2개 이상 보유 상태로 빌런에게 투표한다.
- 리셀러
  - 컨셉: 싸게 사서 비싸게 되파는 사람.
  - 능력: 구매가와 판매가 차이를 추적한다.
  - 미션: 구매가보다 높은 가격으로 2회 재판매한다.

작업:

- `docs/game/rules/jobs.md`를 먼저 작성한다.
- 직업 데이터와 직업 미션 판정 함수를 분리한다.
- 개인 화면에 직업, 능력, 미션 진행도를 표시한다.
- 결과 화면에 직업 미션 달성 여부와 보너스를 표시한다.

완료 기준:

- 문서를 보고 다른 세션이 직업 규칙을 이해할 수 있다.
- 시민도 개인 목표 때문에 의심스러운 행동을 할 동기가 생긴다.
- 직업별 판정 테스트가 있다.

## Issue 4: 다른 와이파이에서도 접속 가능한 배포

제목: `[GAME-4] 다른 와이파이에서도 접속 가능한 배포 경로 확정`

브랜치: `docs/game-4-deployment`

목표:
같은 와이파이가 아닌 친구들도 `/game` 방에 접속할 수 있는 경로를 확정한다.

작업:

- 1차 데모 경로로 Cloudflare Tunnel 사용 절차를 문서화한다.
- `pnpm build`, `pnpm start` 기준 프로덕션 실행을 확인한다.
- Render/Fly 같은 Node 서버 배포 선택지를 비교한다.
- Vercel은 현재 메모리 방 저장소와 맞지 않으므로, 외부 저장소 전환 전에는 주의 사항으로 둔다.

완료 기준:

- 호스트가 로컬 서버와 터널을 켜면 외부 와이파이에서도 접속 가능한 URL을 공유할 수 있다.
- 장기 배포를 위해 필요한 저장소 전환 조건이 문서화된다.

## Issue 5: 평판 시스템 개선

제목: `[GAME-5] 평판 토큰을 신뢰도 시스템으로 재설계`

브랜치: `feat/game-5-reputation`

목표:
"내 평판을 깎아서 남에게 준다"는 구조를 제거하고, 평판을 공개 신뢰도와 사회적 압박 장치로 만든다.

규칙:

- 모든 플레이어는 공개 평판 5로 시작한다.
- 만족 후기는 대상 평판 +1.
- 불만 후기는 대상 평판 -1.
- 후기 작성자는 자기 평판을 지불하지 않는다.
- 평판 0은 즉시 개인 패배가 아니라 조기 시장 마감 또는 강제 투표 진입 조건으로 사용한다.
- 빌런의 평판이 0이 되면 시민 조기 승리.

작업:

- `reputationTokens` 명칭을 `reputation` 또는 동등한 신뢰도 개념으로 바꾼다.
- 후기 처리 로직을 대상 중심으로 바꾼다.
- 악성 후기와 오신고 패널티는 Issue 7의 증거 시스템과 연결한다.

완료 기준:

- 평판이 화폐가 아니라 신뢰도로 작동한다.
- 짧은 게임에서도 평판 변화가 결과에 의미를 갖는다.
- 기존 평판 테스트가 새 규칙으로 갱신된다.

## Issue 6: 물품/카테고리/상태 데이터 확장

제목: `[GAME-6] 물품, 카테고리, 상태 데이터를 거래 추리에 맞게 확장`

브랜치: `feat/game-6-item-data`

목표:
수집가, 거래 품목 공개, 숨은 위험 정보가 작동할 수 있도록 물품 데이터를 확장한다.

작업:

- 물품을 최소 24개 이상으로 확장한다.
- 카테고리는 4개 이상을 유지하고 각 카테고리 물품 수를 균형 있게 맞춘다.
- 각 물품에 `category`, `marketPrice`, `condition`, `riskTag`를 둘 수 있게 한다.
- `acquiredPrice`를 거래 후 보유 카드에 기록할 수 있게 준비한다.

완료 기준:

- 5인 게임에서 카드 분배가 반복적이지 않다.
- 수집가 미션이 억지스럽지 않다.
- 거래 UI가 카테고리와 시세를 표시할 수 있다.

## Issue 7: 빌런 행동 미션과 증거 시스템

제목: `[GAME-7] 빌런 행동 미션과 증거 시스템 추가`

브랜치: `feat/game-7-villain-evidence`

목표:
빌런이 명확히 나쁜 행동을 해야 이기고, 시민은 그 행동을 증거로 추론할 수 있게 한다.

빌런 행동 초안:

- 위험 물건 또는 위장 물건 거래 성공.
- 시세보다 과도하게 비싼 가격으로 판매.
- 거래 취소를 반복 유도.
- 악성 후기로 특정 시민 평판을 낮춤.
- 특정 시민을 투표 대상으로 몰아감.

증거 초안:

- 위험 물건 거래 적발.
- 시세 괴리가 큰 거래 기록.
- 반복 취소 기록.
- 악성 후기 기록.
- 감정으로 확인한 위험 정보.

완료 기준:

- 빌런 미션이 로그와 상태로 판정된다.
- 시민이 토론할 수 있는 증거 단서가 생긴다.
- 신고자 직업이 증거 시스템과 자연스럽게 연결된다.

## Issue 8: 최종 결과 점수표와 승리 조건 정리

제목: `[GAME-8] 최종 결과 점수표와 승리 조건 정리`

브랜치: `feat/game-8-results-scoring`

목표:
팀 승리, 개인 자산, 직업 미션, 빌런 미션을 결과 화면에서 분해해 보여준다.

작업:

- 시민 승리/빌런 승리 기준을 명확히 표시한다.
- 개인 자산, 평판, 직업 보너스, 빌런 미션 보너스를 분해한다.
- 동점 처리 규칙을 정한다.

완료 기준:

- 플레이어가 왜 이겼고 왜 졌는지 결과 화면에서 이해할 수 있다.
- 밸런스 튜닝에 필요한 점수 근거가 남는다.

## Issue 9: 거래 신청을 구매 주도 행동 카드로 재설계

제목: `[GAME-9] 판매자 중심 거래 신청을 구매 주도 행동 카드로 재설계`

브랜치: `feat/game-9-trade-request-action`

목표:
현재처럼 내 턴에 내가 물건을 팔겠다고 신청하는 구조를 고친다. 중고거래의 자연스러운 흐름에 맞게 "거래 신청" 또는 "구매 요청" 행동 카드 중심으로 바꾼다.

규칙 초안:

- 내 턴에 행동 카드를 뽑는다.
- `거래 신청` 카드를 뽑으면 다른 플레이어의 공개 물건 중 하나에 구매 요청을 보낸다.
- 판매자는 수락, 거절, 가격 조정 응답을 할 수 있다.
- 판매자가 먼저 파는 행동은 별도 카드가 있을 때만 가능하게 하거나, 이후 확장으로 둔다.
- 무료나 직거래 같은 특수 카드는 거래 신청의 변형으로 정리한다.

작업:

- `sell` 중심 액션명을 `tradeRequest` 또는 동등한 구매 요청 개념으로 바꾼다.
- 거래 상태의 주도자를 `requesterId`, `ownerId`로 재검토한다.
- UI 문구를 "판매 제안"에서 "거래 신청/구매 요청"으로 바꾼다.
- 기존 거래 테스트를 구매 주도 플로우로 갱신한다.

완료 기준:

- 거래 흐름이 "구매자가 관심 물건에 거래 신청"하는 구조가 된다.
- 판매자와 구매자의 선택지가 현실적인 중고거래 흐름에 맞는다.
- 이후 흥정가 직업과 가격 조정 시스템을 붙이기 쉬워진다.
